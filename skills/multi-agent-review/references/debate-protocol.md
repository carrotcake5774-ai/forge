# Debate Protocol — Multi-Agent Review

## 1. 발언 형식

### 독립 리뷰 (STEP 2)

```markdown
## 독립 리뷰: {role} ({agent_id})

### 강점 분석
1. [ISS-NNN] {강점 설명}
2. ...

### 약점/리스크
1. [ISS-NNN] [{severity}] {약점 설명}
   - 근거: ...
   - 개선 제안: ...

### 구체적 개선 제안
- ...

### 확신도
- [ISS-NNN]: [확정] / [유력] / [확인필요]

### 전체 평가
[강력지지] / [조건부지지] / [재검토필요] / [반대]
- 사유: ...
```

### 토론 발언 (STEP 3, 4)

```markdown
## R{round} 발언: {role} ({agent_id})

### [ISS-NNN]에 대해
- 대상: {target_agent_id}의 주장
- 입장: [동의] / [반론]
- 근거: ...
- (반론 시) 대안: ...

### 새 Issue (발견 시)
- [ISS-NNN] {새 쟁점} — 상태: OPEN
```

### R2 수렴 발언 (STEP 4)

```markdown
## R2 수렴: {role} ({agent_id})

### [ISS-NNN]에 대한 반론 응답
- R1 반론자: {agent_id}
- 응답: [수용] / [유지-재반론]
- (수용 시) 수정 내용: ...
- (유지 시) 재반론 근거: ... (동일 근거 반복 금지)

### Issue 상태 제안
- [ISS-NNN]: RESOLVED — {합의 내용}
- [ISS-NNN]: ESCALATED — {양측 근거 요약}
```

## 2. 심각도 태그

| 태그 | 기준 | 예시 |
|------|------|------|
| `[Critical]` | 문서의 핵심 논리/설계가 잘못됨, 실행 시 심각한 문제 | 아키텍처 결함, 보안 취약점, 논리 모순 |
| `[Major]` | 중요하지만 우회 가능, 품질에 유의미한 영향 | 성능 병목, 확장성 한계, 미고려 엣지케이스 |
| `[Minor]` | 개선하면 좋지만 실행에 지장 없음 | 명명 규칙, 문서 형식, 사소한 누락 |

## 3. 라운드 규칙

### Round 1 (STEP 3)

- **목적**: 교차 검토 — 다른 에이전트의 리뷰에 동의/반론
- **규칙**:
  1. 각 에이전트는 **독립 Task**로 실행 (페르소나 독립성 보장)
  2. 타 에이전트 리뷰를 ISS별로 직접 인용하여 응답
  3. 반론 시 근거 필수 — 감정적/모호한 반론 금지
  4. 새 Issue 발견 시 ISS-NNN 형식으로 추가 가능
  5. 각 ISS에 상태 제안: OPEN -> IN_REVIEW

### Round 2 (STEP 4)

- **목적**: 수렴 — R1 반론에 대한 최종 응답
- **규칙**:
  1. R1에서 **자신에게 제기된 반론**에만 응답
  2. 응답은 반드시 [수용] 또는 [유지-재반론] 중 택일
  3. **동일 근거 반복 금지** — 새로운 논거를 제시하거나 수용
  4. 합의 가능 Issue -> RESOLVED 제안
  5. 합의 불가 Issue -> ESCALATED 제안 + 양측 근거 기록
  6. Devil's Advocate는 추가로 **합의의 블라인드스팟** 최종 점검

### Round 3 (조건부, STEP 4 이후)

- **조건**: R2 후 RESOLVED < 50%일 때만 실행
- **방식**: Opus가 미해결 쟁점을 A안/B안으로 축소하여 제시
- **목적**: 강제 수렴 — 각 에이전트가 A안/B안 중 택일
- **최대 1라운드만 허용**

## 4. 합의 조건

### RESOLVED 판정 기준

Issue가 RESOLVED로 전환되려면:
1. **전원 합의**: 관련 에이전트 전원이 동일 결론에 도달
2. **다수 합의 + 소수 수용**: 과반이 합의하고, 소수가 [수용]으로 응답
3. **근거 기반 수용**: [수용] 시 구체적 수용 사유 명시

### ESCALATED 판정 기준

1. R2까지 합의 불가 + 양측 모두 [유지-재반론]
2. 사실관계 확인이 추가로 필요한 경우
3. 도메인 외 전문가 판단이 필요한 경우

### Devil's Advocate 특별 권한

- 모든 ISS에 대해 발언 가능 (ALL 배정)
- RESOLVED 직전 **블라인드스팟 이의** 1회 가능
- 이의 제기 시 해당 ISS는 IN_REVIEW로 복귀

## 5. 컨텍스트 압축 규칙

토론 라운드 간 산출물 전달 시 Opus가 압축한다.

### 압축 원칙

1. **ISS별 구조 유지** — ISS-NNN 단위로 핵심만 추출
2. **입장 + 근거 보존** — 에이전트의 결론과 핵심 근거 1줄 보존
3. **심각도 태그 보존** — [Critical]/[Major]/[Minor] 유지
4. **직접 인용 보존** — 문서 원문 인용은 축약하되 참조 위치 유지
5. **압축률 목표** — 원문 대비 ~30% (4명 기준 토큰 예산 ~80K 이내)

### 압축 형식

```markdown
## 압축 리뷰: {agent_id} ({role})

평가: [강력지지/조건부지지/재검토필요/반대]

| ISS | 심각도 | 핵심 포인트 | 확신도 |
|-----|--------|------------|--------|
| ISS-001 | Major | {1줄 요약} | 유력 |
| ISS-002 | Minor | {1줄 요약} | 확정 |

주요 개선 제안:
1. {제안 1줄 요약}
2. {제안 1줄 요약}
```

## 6. 발언 제한

| 항목 | 제한 |
|------|------|
| 에이전트당 R1 발언 | ISS별 최대 300자 |
| 에이전트당 R2 발언 | 반론 응답별 최대 200자 |
| 새 Issue 추가 | R1에서만 가능, 에이전트당 최대 2개 |
| 전체 토론 라운드 | 최대 3라운드 (R3는 조건부) |
| Devil 블라인드스팟 이의 | R2에서 1회 |
