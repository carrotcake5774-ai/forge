## Pipeline

```
A → B → C → D
```

순차 실행. 각 단계가 이전 단계의 산출물에 의존한다.

**적용 조건**: 데이터 의존성이 선형적일 때
**장점**: 단순, 디버그 용이, 체크포인트 명확
**단점**: 병렬화 불가, 전체 시간 = 각 단계 합산

**실제 예시** — 통합고용세액공제:
```
parse-agent → step2(분류) → step3(월별 매트릭스) → step4(청년 분류)
  → step5(집계) → step7(유연근무) → report-agent
```

## Fan-out / Fan-in

```
       ┌→ B1 ─┐
  A ──→├→ B2 ─├→ C
       └→ B3 ─┘
```

하나의 단계에서 여러 워커로 분산, 결과를 통합한다.

**적용 조건**: 동일 구조의 독립 입력이 복수 존재할 때
**장점**: 병렬 처리로 시간 단축
**단점**: 부분 실패 처리 필요, 통합 로직 복잡

**병렬 실행 전제 조건** (3개 모두 충족 필수):
1. **입력 독립**: 워커 간 입력 데이터 겹침 없음
2. **파일 충돌 없음**: 동일 파일 동시 쓰기 없음
3. **순서 무관**: 완료 순서가 후속 처리에 영향 없음

**실제 예시** — 바로고 허브별 처리:
```
       ┌→ Sonnet(허브 A) ─┐
Opus ─→├→ Sonnet(허브 B) ─├→ Opus(통합+매핑)
       ├→ Sonnet(허브 C) ─┤
       └→ Sonnet(허브 D) ─┘
```

**부분 실패 처리**:
```
워커 3개 성공 + 1개 실패
  → 성공 결과 보존
  → 실패 워커만 1회 재호출
  → 재호출 실패 → 해당 영역만 에스컬레이션
```

## Hub-Spoke

```
         ┌→ W1 → Opus ─┐
  Opus ─→├→ W2 → Opus ─├→ (검증 후 다음 단계)
         └→ W3 → Opus ─┘
```

Opus가 배분+검증을 모두 담당. Fan-out과 유사하지만 각 워커 결과를 개별 검증한다.

**적용 조건**: 각 워커 결과의 독립 검증이 필요할 때
**장점**: 워커별 품질 관리, 에스컬레이션 세분화
**단점**: Opus 토큰 사용 증가

**실제 예시** — 양도소득세 STEP별 검증:
```
Opus → Sonnet(STEP 1) → Opus(검증) → Sonnet(STEP 2) → Opus(검증) → ...
```

## Conditional Branch

```
        ┌→ [경로 A] → ...
  A → (판단) ─┤
        └→ [경로 B] → ...
```

조건에 따라 다른 워크플로우로 분기한다.

**적용 조건**: 입력 데이터 유형에 따라 처리 방법이 달라질 때
**장점**: 불필요한 단계 건너뛰기 가능
**단점**: 분기 로직의 정확성 필요

**판단 주체**: 항상 Opus (Sonnet은 분기 결정 불가)

**실제 예시** — 양도소득세 STEP 2 조건부:
```
STEP 0 → (물건 복수?) → YES → STEP 2(안분) → STEP 1‖3
                        → NO  → STEP 1‖3 (STEP 2 건너뜀)
```

## Hybrid

실제 프로젝트는 단일 패턴이 아닌 복합 패턴을 사용한다.

**실제 예시** — 양도소득세 전체:
```
[Conditional] STEP 0 → 물건 복수 여부 판단
    │
[Fan-out]     STEP 1 ‖ STEP 3 (병렬)
    │
[Hub-Spoke]   Opus 교차검증 (STEP 1+3 결과)
    │
[Pipeline]    STEP 4 → STEP 5
    │
[Hub-Spoke]   Opus 최종 검증 → Human Review
```

## 패턴 선택 가이드

아래 질문으로 적절한 패턴을 결정한다:

```
Q1. 모든 태스크가 이전 결과에 의존하는가?
  YES → Pipeline
  NO  → Q2

Q2. 동일 구조의 독립 입력이 복수인가?
  YES → Fan-out / Fan-in
  NO  → Q3

Q3. 입력 유형에 따라 처리 방법이 달라지는가?
  YES → Conditional Branch
  NO  → Q4

Q4. 각 워커 결과를 개별 검증해야 하는가?
  YES → Hub-Spoke
  NO  → Pipeline (기본)
```

복합 조건 → Hybrid 구성. 위 질문을 각 도메인/단계별로 반복 적용한다.

## 실행 순서 결정 규칙

1. **의존성 없는 에이전트끼리 병렬 실행** (같은 phase)
2. **의존성 있는 에이전트는 순차 실행** (다른 phase)
3. **검증 에이전트는 검증 대상 완료 후 실행**
4. **오케스트레이터 검증은 모든 병렬 워커 완료 후**
5. **Human Review Gate는 Opus 검증 통과 후**
