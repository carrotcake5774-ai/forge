# Plan Build Process

> Claude × Codex 협업 플랜 빌드 프로세스
> 서로 다른 모델 lineage로 교차검증하는 반복적 정제 워크플로우

---

## AI 역할 정의

| | Claude (Builder) | Codex rescue (Challenger + Creative) |
|---|---|---|
| **강점** | 제약 조건 내 최적 실행, 코드 구체화, 규칙 준수 | 다른 lineage(GPT-5.4) 시각, 비판적 분석, 숨은 결함 탐지 |
| **약점** | 자기가 만든 설계를 자기가 못 깨뜨림 | 기본 동작이 "투입 후 수정" — 리뷰 용도로 쓰려면 프롬프트로 강제 필요 |
| **역할** | Builder + Filter + Enforcer | Creator + Critic (리뷰-only 모드로 사용) |

### Codex rescue 선택 이유

- **OAuth/TOS 이슈 없음** — Claude Code 서브에이전트로 내장
- **Sandbox 내장** — rescue는 격리된 환경에서 실행되어 파일 무단 수정 리스크가 구조적으로 낮음
- **다른 lineage** — GPT-5.4 계열이라 Claude 자기편향을 실제로 깨뜨림

---

## 프로세스 (6 Step)

```
Step 0  Claude: Codex rescue용 브리핑 작성
   ↓
Step 1  Codex rescue: 플랜 초안 작성 (창조)
   ↓
Step 2  Claude: 1차 가공 (정제 + 구체화)
   ↓
Step 3  Claude: ABC 리뷰 프롬프트 생성 ──→ Codex rescue A/B/C 직렬 3회 호출
   ↓                                          ↓
Step 4                                   Codex rescue: ABC 리뷰 (리포트만)
   ↓                                          ↓
Step 5  Claude: 필터 + 반영
   ↓
   ↻  Step 3~5 반복 (N회, 보통 2~3회면 수렴)
   ↓
Step 6  최종 플랜 확정
```

---

## Step 0 — Claude: 브리핑 작성

Codex rescue에게 전달할 "서비스 분석 브리핑"을 Claude가 작성한다.

**포함 내용:**
- 프로젝트 개요 (도메인, 배포 형태)
- 현재 코드 상태 (모델 수, 서비스 수, 라우터 수, 프론트엔드 페이지 수)
- 기술 스택 요약
- **변경 불가 제약 (IMMUTABLE CONSTRAINTS)** — 호스트 프로젝트 CLAUDE.md의 DO NOT 규칙 전문
- 기존 아키텍처 결정사항 요약
- 이번 플랜의 목표/범위

**핵심:** "변경 불가 제약"을 별도 섹션으로 명확히 분리하여 외부 모델이 무시하지 못하게 강조.

```markdown
## ⛔ IMMUTABLE CONSTRAINTS (위반 시 제안 자체가 기각됨)
- (호스트 프로젝트 CLAUDE.md의 DO NOT 규칙 전문)

위 제약을 위반하는 제안은 하지 마시오.
```

---

## Step 1 — Codex rescue: 플랜 초안 작성 (창조)

Claude가 `codex:rescue`를 호출하여 초안 작성을 위임한다.

**호출 형태:**

```
codex:rescue 서브에이전트에 다음 브리핑을 전달하고 플랜 초안 작성을 요청:

[브리핑 내용 전문]

요청사항:
1. 도메인의 실무 워크플로우를 분석하여 현재 서비스에 빠진 기능을 발견하세요.
2. 경쟁 서비스의 장단점을 분석하세요.
3. 기능별 우선순위와 구현 순서를 제안하세요.
4. 각 기능의 UI/UX 와이어프레임을 텍스트로 스케치하세요.

⛔ IMMUTABLE CONSTRAINTS 섹션의 규칙을 위반하는 제안은 금지합니다.

⛔ 출력 규약: 플랜 초안만 stdout에 마크다운으로 작성하시오. 실제 파일 생성·수정은 금지.
```

**기대 산출물:** 다른 lineage 시각의 창의적 아이디어 + 실무 인사이트 + 경쟁사 벤치마크

**주의:** Codex rescue의 기본 동작은 "조사→진단→수정"이므로, 프롬프트 말미에 **파일 수정 금지 + stdout만** 을 반드시 강조한다. (feedback_codex_review_only 참조)

---

## Step 2 — Claude: 1차 가공 (정제)

Codex rescue의 초안을 Claude가 가공:

**제거:**
- CLAUDE.md DO NOT 규칙 위반 항목
- 규제상 불가능한 항목
- 이미 구현 완료된 기능
- 기존 아키텍처 결정과 충돌하는 항목

**변환:**
- 추상적 아이디어 → 구체적 모델/서비스/API/라우터 설계
- 비현실적 범위 → 실현 가능한 Phase로 분할
- UI 스케치 → 프로젝트 디자인 시스템 기반 컴포넌트 명세

**보강:**
- 패키지 배치 맵 (어느 패키지에 위치하는지)
- 의존성 방향 검증
- 테스트 전략 + 성능 벤치마크

## Step 3 — Claude: ABC 리뷰 프롬프트 생성

1차 가공된 플랜에 대해 Codex rescue용 리뷰 프롬프트를 3종류(A/B/C)로 생성.

### 프롬프트 생성 규칙

1. **자기완결형 (Self-contained):** 각 프롬프트는 공통 브리핑 + 개별 질문이 합쳐진 완성본. 한 번의 rescue 호출로 전달 가능해야 함
2. **1 프롬프트 = 1 파일:** `.claude/prompts/{플랜명}-{버전}/input/{Type}{번호}_{topic}.md` 형식으로 개별 파일 생성
3. **공통 브리핑 매번 포함:** "아까 보낸 거 참고해" 금지 — rescue가 세션 간 컨텍스트를 공유한다고 가정하지 말 것
4. **각 프롬프트는 별도 `codex:rescue` 직렬 호출:** **병렬 호출 금지** (rescue 런타임 상태 공유 가능성)
5. **IMMUTABLE CONSTRAINTS 전문을 매번 삽입:** 축약하거나 "위와 동일" 참조 금지
6. **Type A 파괴의 경우 "새로운 기능을 제안하지 말고" 범위 제한 명시**
7. **Type B/C는 창의 범위를 열어두되, CONSTRAINTS만 울타리로 유지**
8. **산출물 작성 규칙을 프롬프트 마지막에 반드시 배치:** 응답이 일관된 마크다운 형식(ID 채번 + 심각도/우선순위 + 요약 테이블)을 따르도록 강제. Claude가 output을 파싱할 때 구조화된 데이터로 바로 처리 가능
9. **리뷰-only 강제 필수 (2중 방어):**
   - **프롬프트 첫 줄**: `리뷰 모드: 리포트만 작성, 파일·플랜 수정 절대 금지`
   - **프롬프트 말미**: `원본 플랜의 어떤 부분도 수정하지 마시오. 리뷰 의견만 별도 섹션(## Findings, ## Risks, ## Recommendations)으로 작성하시오.`
   - rescue 응답에 수정 diff나 패치 제안이 포함되면 해당 라운드는 **전체 폐기** 후 재요청

### Type A — 파괴 (비판적 사고)

목적: 기존 설계의 기술적 결함 탐지. 새 기능 제안 금지.

```
리뷰 모드: 리포트만 작성, 파일·플랜 수정 절대 금지

[공통 브리핑 전문]

## 리뷰 유형: Type A — 파괴
## 리뷰 대상
[특정 Feature 설계]

## 질문
[구체적 파괴 질문 — 동시성, 장애, 엣지 케이스 등]

새로운 기능을 제안하지 말고, 기존 설계의 구멍만 지적해주세요.

원본 플랜의 어떤 부분도 수정하지 마시오.
리뷰 의견만 별도 섹션(## Findings, ## Risks, ## Recommendations)으로 작성하시오.
```

### Type B — 발견 (창의적 사고)

목적: 우리가 모르는 실무 워크플로우/고충 발견.

```
리뷰 모드: 리포트만 작성, 파일·플랜 수정 절대 금지

[공통 브리핑 전문]

## 리뷰 유형: Type B — 발견
## 현재 구현 예정 기능
[기능 목록]

## 질문
[열린 질문 — 실무 패턴, 경쟁사 불만, 누락 기능]

⛔ IMMUTABLE CONSTRAINTS를 위반하는 제안은 금지합니다.

원본 플랜의 어떤 부분도 수정하지 마시오.
리뷰 의견만 ## Findings, ## Risks, ## Recommendations 섹션으로 작성하시오.
```

### Type C — 디자인 (UI/UX 구성)

목적: 경쟁사 참고 + 현대 SaaS 스타일의 UI/UX 레이아웃 제안.

```
리뷰 모드: 리포트만 작성, 파일·플랜 수정 절대 금지

[공통 브리핑 전문]

## 리뷰 유형: Type C — 디자인
## 디자인 시스템
[현재 디자인 시스템 명세]

## 리뷰 대상 화면
[화면 명세]

## 질문
[UI/UX 제안 요청 — 경쟁사 참고, 와이어프레임, 정보 계층]

원본 플랜의 어떤 부분도 수정하지 마시오.
리뷰 의견만 ## Findings, ## Risks, ## Recommendations 섹션으로 작성하시오.
```

### 디렉토리 구조

```
.claude/prompts/{플랜명}-{버전}/
├── input/                  ← Claude가 생성, codex:rescue 직렬 호출
│   ├── A1_{topic}.md
│   ├── A2_{topic}.md
│   ├── B1_{topic}.md
│   └── C1_{topic}.md
└── output/                 ← Claude가 rescue 응답을 저장 후 필터링
    ├── A1_{topic}.md
    └── B1_{topic}.md
```

**리뷰 회차 = 플랜 버전.** 리뷰 반영 후 플랜 버전이 올라가므로 라운드 폴더 불필요.
- v1.3 리뷰 → 반영 → v1.4 생성
- v1.4 리뷰 → 반영 → v1.5 (수렴 시 FINAL)

**네이밍 규칙:**
- 플랜명: kebab-case (예: `mega-evolution`)
- 버전: 리뷰 시작 시점의 플랜 버전 (예: `v1.3`)
- 파일명: `{Type}{번호}_{topic}.md` (예: `A1_audit_cache.md`)

**워크플로우 (rescue 직렬 자동화):**
1. Claude가 `{플랜명}-{버전}/input/`에 프롬프트 파일 생성
2. Claude가 각 input 파일을 `codex:rescue`에 **직렬**로 전달 (A → B → C 순):
   - Round A 완료 → Round B 시작 → Round C 시작 (병렬 X)
   - 각 호출의 응답을 `output/{파일}.md`에 저장
3. Claude가 rescue 응답을 검증:
   - 응답에 수정 diff/패치 제안 포함 → **해당 라운드 전체 폐기** 후 재요청
   - 응답이 리뷰 리포트 형식(## Findings / ## Risks / ## Recommendations) 준수 → 유지
4. Claude가 `output/*.md` 필터링 + 플랜 반영 → 버전업
5. 다음 리뷰 필요 시 → 새 버전 폴더에 input 생성 → 반복

## Step 4 — Codex rescue: ABC 리뷰 실행 (직렬 3회)

Claude가 각 프롬프트 파일을 `codex:rescue` 서브에이전트에 **직렬로** 3회 전달한다.

```
Round A: codex:rescue with A1_*.md → output/A1_*.md 수신
   ↓ (이전 라운드 완료 대기)
Round B: codex:rescue with B1_*.md → output/B1_*.md 수신
   ↓ (이전 라운드 완료 대기)
Round C: codex:rescue with C1_*.md → output/C1_*.md 수신
```

**핵심 규칙:**
- **직렬 호출** — rescue 런타임 상태 공유 우려로 병렬 호출 금지
- **독립 브리핑** — 각 호출은 자체 브리핑을 포함 (컨텍스트 격리)
- **응답 검증** — 수정 제안 포함 시 즉각 폐기 + 재호출
- **Sandbox 신뢰** — rescue가 sandbox에서 실행되므로 파일 무단 수정 리스크는 구조적으로 낮음 (이전 세대 CLI 워크플로우의 3중 방어가 불필요)

**과거 사고 참고 (이전 세대 CLI 워크플로우):**
외부 CLI 기반 리뷰 워크플로우에서 "제안해줘"를 "구현해줘"로 오인해 파일을 무단 수정한 사고가 있었다. 이에 read-only 플래그 + 프롬프트 강제 + git-guard의 3중 방어를 사용했다. 현재는 Codex rescue의 sandbox와 "리뷰-only 강제" 프롬프트로 같은 리스크를 처리한다.

---

## Step 5 — Claude: 필터 + 반영

Codex rescue 리뷰 결과를 분류:

| 분류 | 기준 | 처리 |
|------|------|------|
| **반영** | 유의적 + 실현 가능 + 제약 준수 | 플랜에 반영 |
| **경미 반영** | 유의적이나 소규모 | 한 줄 추가 또는 경고 수준 |
| **기각** | 제약 위반 또는 비현실적 | 사유와 함께 기각 기록 |
| **보류** | 현재 scope 밖이나 향후 가치 | "향후 고려" 섹션에 기록 |

모든 판단을 플랜의 "리뷰 반영 이력" 부록에 기록하여 추적 가능하게 유지.

---

## Step 3~5 반복 기준

| 회차 | 초점 | 수렴 판단 |
|------|------|----------|
| 1회 | 큰 구조적 결함 + 누락 기능 발견 | 신규 반영 5건 이상이면 2회 진행 |
| 2회 | 세부 설계 결함 + UI/UX 정교화 | 신규 반영 3건 이하면 수렴 |
| 3회 | 최종 엣지 케이스 + 보안 | 신규 반영 0건이면 확정 |

보통 **2~3회면 수렴**. 4회 이상은 과잉.

---

## Step 6 — 최종 플랜 확정

Claude가 최종 문서를 정리:
- 모든 리뷰 반영 이력 부록 포함
- 기각 항목 + 사유 기록
- 버전 넘버링 (v1.0 → v1.1 → ... → vFINAL)

---

## 주의사항

### Codex rescue 사용 규칙

rescue는 원래 "투입해서 문제를 해결"하는 용도이므로 기본 동작은 조사→진단→수정이다. 플랜 리뷰 용도로 쓸 때는 **리뷰-only 모드**로 강제해야 한다.

| 계층 | 방어 수단 | 역할 |
|---|---|---|
| 1. Sandbox | rescue 내장 sandbox | 파일 수정이 격리 환경에만 국한 |
| 2. 프롬프트 강제 (첫 줄) | "리뷰 모드: 리포트만 작성, 파일·플랜 수정 절대 금지" | LLM 의도 수준에서 행동 제약 |
| 3. 프롬프트 강제 (말미) | "원본 플랜의 어떤 부분도 수정하지 마시오. 리뷰 의견만 ## Findings / ## Risks / ## Recommendations 로 작성" | 응답 형식 강제 |
| 4. 응답 검증 | 수정 diff/패치 포함 시 전체 폐기 | 우회 응답 필터 |

**필수 규칙:**
- **리뷰-only 강제 문구 생략 금지** — 첫 줄 + 말미 2곳 모두 필수
- **병렬 호출 금지** — A/B/C는 반드시 직렬 실행 (rescue 런타임 상태 공유 우려)
- **수정 제안 포함 응답 폐기** — 필터링 대상이 아니라 전체 폐기 대상

### Codex rescue에게 절대 하지 말 것
- "이 플랜 전체를 리뷰해줘" → 일반론 폭주
- 제약 조건 없이 열린 질문 → 룰브레이킹 제안 범람
- 병렬 호출 → 컨텍스트 오염 리스크

### Codex rescue에게 잘 시키는 법
- **좁은 범위 + 구체적 질문** (Feature 단위, 화면 단위)
- **IMMUTABLE CONSTRAINTS를 매번 첨부** (생략하면 위반 제안 발생)
- **UI/UX 제안 시 경쟁사 화면을 참고하라고 명시**
- **리뷰-only 강제 문구를 처음과 끝에 배치**

### Claude가 주의할 것
- Codex 초안의 좋은 아이디어를 "규칙 위반"으로 과잉 기각하지 않기
- 위반 부분만 잘라내고 핵심 인사이트는 살리기
- 기각 시 "왜 기각했는지 + 대안은 없는지" 한 줄 남기기
- rescue가 리뷰-only 모드를 벗어난 응답을 내면, 프롬프트 강제 문구가 충분했는지 재점검
