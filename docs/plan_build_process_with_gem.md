# Plan Build Process (with Gemini)

> Claude × Gemini 협업 플랜 빌드 프로세스
> 각 AI의 강점을 활용한 반복적 정제 워크플로우

---

## AI 역할 정의

| | Claude (Blue Team) | Gemini (Red Team + Creative) |
|---|---|---|
| **강점** | 제약 조건 내 최적 실행, 코드 구체화, 규칙 준수 | 웹서치 기반 창의적 사고, 비판적 분석, UI/UX 감각 |
| **약점** | 자기가 만든 설계를 자기가 못 깨뜨림 | 제약 조건 무시 경향, 일반론 폭주 |
| **역할** | Builder + Filter + Enforcer | Creator + Challenger + Designer |

---

## 프로세스 (6 Step)

```
Step 0  Claude: Gemini용 브리핑 작성
   ↓
Step 1  Gemini: 플랜 초안 작성 (창조)
   ↓
Step 2  Claude: 1차 가공 (정제 + 구체화)
   ↓
Step 3  Claude: ABC 리뷰 프롬프트 생성 ──→ Gemini 실행
   ↓                                          ↓
Step 4                                   Gemini: ABC 리뷰
   ↓                                          ↓
Step 5  Claude: 필터 + 반영
   ↓
   ↻  Step 3~5 반복 (N회, 보통 2~3회면 수렴)
   ↓
Step 6  최종 플랜 확정
```

## Step 0 — Claude: 브리핑 작성

Gemini에게 전달할 "서비스 분석 브리핑"을 Claude가 작성한다.

**포함 내용:**
- 프로젝트 개요 (K-GAAP 회계 SaaS, Lite/Pro 이중 배포)
- 현재 코드 상태 (모델 수, 서비스 수, 라우터 수, 프론트엔드 페이지 수)
- 기술 스택 요약
- **변경 불가 제약 (IMMUTABLE CONSTRAINTS)** — CLAUDE.md DO NOT 규칙 전문
- 기존 아키텍처 결정사항 요약
- 이번 플랜의 목표/범위

**핵심:** "변경 불가 제약"을 별도 섹션으로 명확히 분리하여 Gemini가 무시하지 못하게 강조.

```markdown
## ⛔ IMMUTABLE CONSTRAINTS (위반 시 제안 자체가 기각됨)
- 서버에 AI/ML 로직 탑재 금지. AI는 CLI에서만 실행
- FLOAT 금지. 금액은 반드시 NUMERIC(18,2)
- ez_hr → ez_accounting 직접 import 금지
- ... (CLAUDE.md DO NOT 규칙 전문)

위 제약을 위반하는 제안은 하지 마시오.
```

## Step 1 — Gemini: 플랜 초안 작성 (창조)

Gemini에게 브리핑 + 아래 지시를 전달:

```
당신은 K-GAAP 회계 SaaS의 전략 기획자입니다.

[브리핑 내용 붙여넣기]

위 서비스에 대해 다음을 포함하는 플랜 초안을 작성하세요:

1. 한국 세무사무실/회계사무실의 실무 워크플로우를 웹서치하여
   현재 서비스에 빠진 기능을 발견하세요.
2. 더존, 세무사랑, WEHAGO 등 경쟁 서비스의 장단점을 분석하세요.
3. 기능별 우선순위와 구현 순서를 제안하세요.
4. 각 기능의 UI/UX 와이어프레임을 텍스트로 스케치하세요.

⛔ IMMUTABLE CONSTRAINTS 섹션의 규칙을 위반하는 제안은 금지합니다.
```

**기대 산출물:** 창의적 아이디어 + 실무 인사이트 + 경쟁사 벤치마크 (+ 일부 망상)

Gemini 초안을 Claude가 가공:

**제거:**

**변환:**

## Step 3 — Claude: ABC 리뷰 프롬프트 생성

1차 가공된 플랜에 대해 Gemini용 리뷰 프롬프트를 3종류로 생성.

### 프롬프트 생성 규칙

1. **자기완결형 (Self-contained):** 각 프롬프트는 공통 브리핑 + 개별 질문이 합쳐진 완성본. `gemini -p "$(cat {file})"` 한 번 호출로 전달 가능해야 함
2. **1 프롬프트 = 1 파일:** `.claude/prompts/gemini_{type}_{id}_{topic}.md` 형식으로 개별 파일 생성
3. **공통 브리핑 매번 포함:** "아까 보낸 거 참고해" 금지. Gemini가 기억을 지어내는 것 방지
4. **각 프롬프트는 별도 `gemini` CLI 호출로 실행:** 호출 간 컨텍스트 격리로 세션 오염 차단
5. **IMMUTABLE CONSTRAINTS 전문을 매번 삽입:** 축약하거나 "위와 동일" 참조 금지
6. **프롬프트 끝에 "새로운 기능을 제안하지 말고" 같은 범위 제한을 명시** (Type A 파괴의 경우)
7. **Type B/C는 창의 범위를 열어두되, CONSTRAINTS만 울타리로 유지**
8. **산출물 작성 규칙을 프롬프트 마지막에 반드시 배치:** Gemini 응답이 일관된 마크다운 형식(ID 채번 + 심각도/우선순위 + 요약 테이블)을 따르도록 강제. 이를 통해 Claude가 output을 파싱할 때 구조화된 데이터로 바로 처리 가능
9. **플랜 수정 금지 지시 필수:** 프롬프트 말미에 "원본 플랜 수정 금지. 리뷰 리포트만 stdout 출력. 파일 생성/수정 금지." 강제 문구 삽입 — Gemini가 "리뷰"를 "개선 작업"으로 해석해 파일을 수정하는 경향에 대한 이중 방어 (`--approval-mode plan` + 프롬프트 강제 + git-guard의 3중 방어 중 2단계)

### Type A — 파괴 (비판적 사고)

목적: 기존 설계의 기술적 결함 탐지. 새 기능 제안 금지.

```
[공통 브리핑 전문]

## 리뷰 유형: Type A — 파괴
## 리뷰 대상
[특정 Feature 설계]

## 질문
[구체적 파괴 질문 — 동시성, 장애, 엣지 케이스 등]

새로운 기능을 제안하지 말고, 기존 설계의 구멍만 지적해주세요.
```

### Type B — 발견 (웹서치 기반 창의적 사고)

목적: 우리가 모르는 실무 워크플로우/고충 발견. 웹서치 명시 요청.

```
[공통 브리핑 전문]

## 리뷰 유형: Type B — 발견
## 현재 구현 예정 기능
[기능 목록]

## 질문
[웹서치 기반 열린 질문 — 실무 패턴, 경쟁사 불만, 누락 기능]

⛔ IMMUTABLE CONSTRAINTS를 위반하는 제안은 금지합니다.
단, "서버가 아닌 CLI에서 AI로 처리할 수 있는 워크플로우"는 제안 가능합니다.
```

### Type C — 디자인 (UI/UX 구성)

목적: 경쟁사 참고 + 현대 SaaS 스타일의 UI/UX 레이아웃 제안.

```
[공통 브리핑 전문]

## 리뷰 유형: Type C — 디자인
## 디자인 시스템
[현재 디자인 시스템 명세]

## 리뷰 대상 화면
[화면 명세]

## 질문
[UI/UX 제안 요청 — 경쟁사 웹서치 참고, 와이어프레임, 정보 계층]
```

### 디렉토리 구조

```
.claude/prompts/{플랜명}-{버전}/
├── input/                  ← Claude가 생성, `gemini -p`로 자동 호출
│   ├── A1_{topic}.md
│   ├── A2_{topic}.md
│   ├── B1_{topic}.md
│   └── C1_{topic}.md
└── output/                 ← Claude가 stdout 리다이렉트로 자동 저장 후 필터링
    ├── A1_{topic}.md
    └── B1_{topic}.md
```

**리뷰 회차 = 플랜 버전.** 리뷰 반영 후 플랜 버전이 올라가므로 라운드 폴더 불필요.
- v1.3 리뷰 → 반영 → v1.4 생성
- v1.4 리뷰 → 반영 → v1.5 (수렴 시 FINAL)

**네이밍 규칙:**
- 플랜명: kebab-case (예: `mega-evolution`)
- 버전: 리뷰 시작 시점의 플랜 버전 (예: `v1.3`)
- 파일명: `{Type}{번호}_{topic}.md` (예: `A1_audit_cache.md`)

**워크플로우 (CLI 자동화):**
1. Claude가 `{플랜명}-{버전}/input/`에 프롬프트 파일 생성
2. Claude가 `Bash` 툴로 각 input 파일을 `gemini` CLI에 자동 호출:
   - `gemini -p "$(cat input/{파일}.md)" --approval-mode plan -o text > output/{파일}.md`
   - 호출 전후 `git status --porcelain` 비교로 수정 감지 → 차이 시 rollback + 해당 응답 폐기
3. Claude가 `output/*.md` 필터링 + 플랜 반영 → 버전업
4. 다음 리뷰 필요 시 → 새 버전 폴더에 input 생성 → 반복

## Step 4 — Gemini: ABC 리뷰 실행 (CLI 자동화)

Claude가 `Bash` 툴로 각 프롬프트 파일을 `gemini` CLI에 자동 전달합니다.

```bash
# 1. 호출 전 git 상태 스냅샷
BEFORE=$(git status --porcelain)

# 2. 프롬프트 실행 (read-only 모드 필수)
gemini -p "$(cat .claude/prompts/{플랜명}-{버전}/input/A1_topic.md)" \
  --approval-mode plan \
  -o text \
  > .claude/prompts/{플랜명}-{버전}/output/A1_topic.md

# 3. 호출 후 git 상태 재확인
AFTER=$(git status --porcelain)

# 4. 수정 감지 시 rollback + 응답 폐기
if [ "$BEFORE" != "$AFTER" ]; then
    git checkout -- .
    rm .claude/prompts/{플랜명}-{버전}/output/A1_topic.md
    echo "⚠️ Gemini가 파일을 수정함. 응답 폐기됨."
fi
```

**CLI 옵션 해설:**
- `--approval-mode plan`: **read-only 모드 필수** — Gemini 자체의 파일 수정 권한 차단 (3중 방어의 1단계)
- `-o text`: stdout 파싱 용이
- 각 프롬프트는 **독립 호출** — 세션 간 컨텍스트 오염 차단 (기억 지어내기 방지)

**병렬 실행 가능:** 프롬프트 간 의존성이 없으므로 여러 input 파일을 백그라운드로 동시 호출해도 무방. 단, rate limit 주의.

## Step 5 — Claude: 필터 + 반영

Gemini 리뷰 결과를 분류:

| 분류 | 기준 | 처리 |
|------|------|------|
| **반영** | 유의적 + 실현 가능 + 제약 준수 | 플랜에 반영 |
| **경미 반영** | 유의적이나 소규모 | 한 줄 추가 또는 경고 수준 |
| **기각** | 제약 위반 또는 비현실적 | 사유와 함께 기각 기록 |
| **보류** | 현재 scope 밖이나 향후 가치 | "향후 고려" 섹션에 기록 |

모든 판단을 플랜의 "리뷰 반영 이력" 부록에 기록하여 추적 가능하게 유지.

## Step 3~5 반복 기준

| 회차 | 초점 | 수렴 판단 |
|------|------|----------|
| 1회 | 큰 구조적 결함 + 누락 기능 발견 | 신규 반영 5건 이상이면 2회 진행 |

## 주의사항

### Gemini 실행 환경 규칙 (CLI 자동화 + 3중 방어)

리뷰 프롬프트는 `gemini` CLI로 자동 실행합니다. 과거 "CLI 금지" 규칙은 폐기되었으며, 현재는 3중 방어 메커니즘으로 파일 수정 사고를 방지합니다.

| 계층 | 방어 수단 | 역할 |
|---|---|---|
| 1. CLI 플래그 | `--approval-mode plan` | Gemini 자체 read-only 모드로 파일 수정 원천 차단 |
| 2. 프롬프트 강제 | "수정 금지, 리포트만 stdout 출력" 문구 | LLM 의도 수준에서 행동 제약 |
| 3. git-guard | 호출 전후 `git status --porcelain` 비교 | 앞선 2단계를 우회한 수정을 외부에서 감지 + rollback |

**필수 규칙:**
- **`--approval-mode plan` 생략 금지** — 다른 approval 모드(`default`, `auto_edit`, `yolo`)는 파일 수정을 허용하므로 리뷰 용도로 부적합
- **프롬프트 말미에 수정 금지 강제 문구 필수** — 프롬프트 생성 규칙 #9 참조
- **git-guard 우회 금지** — Bash로 호출 시 반드시 before/after 스냅샷 비교 수행
- **Git worktree 격리는 선택사항** — `gemini -w`로 worktree 내에서 실행하면 추가 격리 가능 (대규모 리뷰 시 권장)

**과거 사고 기록 (CLI 금지 규칙 시절):**
"제안해줘" 명령을 Gemini가 "구현해줘"로 오인해 Sidebar 수정 + 방향이 다른 Admin 대시보드 무단 생성한 사고가 발생했음. 당시 대응은 "CLI 자체 사용 금지"였으나, 현재는 `--approval-mode plan` + 프롬프트 강제 + git-guard 조합으로 사고를 구조적으로 차단하므로 CLI 자동화가 안전하게 가능.

**코딩 작업용 Gemini 사용:**
Gemini를 **코딩 에이전트**로 사용하는 것은 여전히 별도 영역. 리뷰 파이프라인과 분리하여, Claude가 구체적 구현 스펙을 확정한 후 단순 반복 코딩 작업에만 제한적으로 사용 가능.

### Gemini에게 절대 하지 말 것
- "이 플랜 전체를 리뷰해줘" → 일반론 폭주
- 제약 조건 없이 열린 질문 → 룰브레이킹 제안 범람
- 코드 에디터 환경에서 리뷰 프롬프트 실행 → 무단 코드 생성

### Gemini에게 잘 시키는 법
- **좁은 범위 + 구체적 질문** (Feature 단위, 화면 단위)
- **웹서치를 명시적으로 요청** ("웹서치해서 실무 사례를 찾아줘")
- **IMMUTABLE CONSTRAINTS를 매번 첨부** (생략하면 위반 제안 발생)
- **UI/UX 제안 시 경쟁사 화면을 참고하라고 명시**
